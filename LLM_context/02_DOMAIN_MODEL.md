# E-MindBridge — Domain Model Summary

## Entities & Responsibilities

### `auth.users` (Supabase-managed)
The authentication source of truth. Never written to directly. All application logic references `profiles` instead.

---

### `profiles`
The central entity. Every user — patient, counselor, or admin — has exactly one profile linked to `auth.users`. Stores display information, role, org/school context, and anonymity preference.

**Key fields:** `role`, `full_name`, `school_or_org`, `is_active`, `is_anonymous`

---

### `counselor_profiles`
A **one-to-one extension** of `profiles` for users with `role = 'counselor'`. Stores professional metadata that only counselors have.

**Key fields:**
- `specialties TEXT[]` — array of `primary_category` labels the counselor is trained in (e.g. `{academic, grief, career}`). Used for smart patient assignment.
- `is_available BOOLEAN` — availability flag toggled by the counselor or set by the system during active sessions. Drives auto-assignment logic.
- `max_patient_load INTEGER` — soft cap on concurrent patient assignments.
- `license_number`, `years_of_experience` — professional credentials.

---

### `consent_records`
A versioned record of a user's agreement to the Ethics & Privacy module. A user without a current `status = 'agreed'` record for the latest `consent_version` is blocked from the dashboard. Immutable — no delete policy.

---

### `mood_logs`
One entry per patient per calendar day. Enforced via a unique index on `(user_id, logged_at::DATE)`. The mood value feeds both AI analysis and aggregate analytics.

---

### `journal_entries`
Free-text journal entries written by patients. Encrypted at rest. The `is_analyzed` flag is `FALSE` on insert and flipped to `TRUE` after the AI Edge Function processes it. Counselors have no RLS access to this table's raw content.

---

### `ai_reports`
The core analytical artifact. Can be generated two ways:
1. **Automatically** — triggered by a new `journal_entry` save.
2. **On-request** — triggered by an approved `report_requests` record.

Contains the full MindProfile: risk level, sentiment, primary category, longitudinal pattern, emotional markers, threat detection flags, and AI recommendations. Referenced by evaluations, sessions, and notifications.

---

### `report_requests`
The **admin approval gate** for counselor-initiated AI reports. A counselor submits a request with a patient ID and date range. Admin reviews and sets `status` to `approved` or `rejected`. On approval, a database trigger or Edge Function webhook fires the AI generation pipeline.

**States:** `pending → approved → fulfilled` or `pending → rejected`

---

### `counselor_evaluations`
The human-in-the-loop layer. After an AI report is generated, a counselor must submit an evaluation: clinical impression, recommendation status (approved / modified / rejected), and a chosen next step. This record is what drives scheduling decisions.

---

### `counseling_sessions`
A scheduled or completed tele-counseling session between one patient and one counselor. Can be proactive (patient-initiated) or reactive (counselor-scheduled). Linked optionally to an AI report and an evaluation. Tracks room URL, bandwidth mode, session notes, and timestamps.

---

### `counselor_patient_assignments`
A **many-to-many join** between counselors and patients. Supports multiple counselors per patient (e.g. primary + backup). Tracks whether the assignment is active and how it was created (auto-assigned vs admin-assigned). Patients default to keeping their first counselor but can request reassignment.

**Key fields:** `assignment_type` (`auto`, `manual`, `patient_requested`), `is_primary`, `is_active`

---

### `report_requests`
Submitted by counselors to request an AI report for a specific patient over a date range. Reviewed by admin. Creates an audit trail of every AI report generation event that wasn't system-triggered.

---

### `notifications`
In-app alerts delivered to a specific recipient. Can be system-generated (null sender) or counselor-to-patient. Linked to a session or report for deep-linking in the UI.

---

### `analytics_snapshots`
Pre-aggregated daily statistics scoped to a counselor's patient group. Generated by a scheduled cron job (Supabase `pg_cron` or external scheduler). Provides the macro-view data for the counselor dashboard without expensive real-time queries.

---

### `wellness_quotes`
Curated quotes shown on the patient dashboard. Tagged with `primary_category` values and `mood_level` targets so the `get_quotes_for_patient()` function can return contextually relevant content based on the patient's latest AI report category and today's mood.

---

### `wellness_resources`
Self-help modules (breathing exercises, CBT worksheets, articles) available via the Wellness Compass. Referenced by AI recommendations.

---

### `audit_logs`
Append-only table. Records every sensitive action: report views, AI triggers, session scheduling, admin approvals/rejections. Never deleted. Admins have full read access; users can see their own entries only.

---

## Key Relationships (Plain English)

- One **auth user** → one **profile** → optionally one **counselor_profile**
- One **patient** → many **mood_logs** (one per day)
- One **patient** → many **journal_entries** → each → one **ai_report**
- One **counselor** → many **report_requests** → reviewed by **admin** → triggers **ai_report**
- One **ai_report** → one **counselor_evaluation** → drives **counseling_session** scheduling
- One **patient** ↔ many **counselors** via **counselor_patient_assignments** (one is primary)
- One **counseling_session** → one **notification** to each participant
- One **counselor** → **analytics_snapshots** scoped to their patient group
- One **patient** → **wellness_quotes** filtered dynamically by mood + AI report category
